# VOC Analysis 리포트 명세

> AI팀 개발 핸드오프용. **어떤 VOC 데이터가 들어와도 동일한 7개 슬롯**으로 구성되는 범용 리포트.
>
> 본 문서는 입력 데이터 정의(§1)부터 출력 리포트 골격(§3~§9)까지 단일 진입점으로 통합되어 있습니다.
>
> 관련 파일:
> - 비즈니스 로직 (소스별 분석 패턴) → [skills/voc-analysis.md](../../skills/voc-analysis.md)
> - 컴포넌트 데이터 스키마 → [src/lib/constants.ts](../../../lib/constants.ts) `COMPONENT_DATA_SCHEMAS`
> - 콘텐츠 규칙 (두괄식 등) → [sample-generator.md](../../sample-generator.md), [strategy-writer.md](../../strategy-writer.md)
> - 전처리 출력 타입 → [orchestrator/types.ts](../../orchestrator/types.ts) `VocPreprocessOutput`

## 1. 입력 데이터

### 1.1 입력 형식
CSV 또는 JSON. 1행 = 1건의 고객 피드백.

### 1.2 지원 데이터 소스
- 앱 스토어 리뷰 (Google Play, App Store)
- NPS/CSAT 서술형 응답
- CS 티켓/채팅 텍스트
- 설문조사 주관식
- B2B 제품 피드백 (인앱, 기능 요청)
- 소셜 미디어 / 커뮤니티 멘션

### 1.3 컬럼 정의

#### 필수 (없으면 리포트 생성 불가)

| 컬럼 | 타입 | 설명 | 예시 |
|------|------|------|------|
| `content` | text | 고객이 작성한 피드백 원문 | "릴스 넘기다 보면 화면 갑자기 밝아져" |

#### 준필수 — 감성 분류에 필요 (둘 중 하나 이상)

| 컬럼 | 타입 | 설명 | 없을 때 처리 |
|------|------|------|-------------|
| `score` | integer (1-5) | 별점 또는 만족도 점수 | 텍스트 기반 감성 자동 분류 |
| `nps_score` | integer (0-10) | NPS 점수 | NPS 분석 제외, score 또는 텍스트 감성 사용 |
| `sentiment` | string (positive/neutral/negative) | 사전 분류된 감성 | score 기반 자동 분류 (4-5=긍정, 3=중립, 1-2=부정) |

> score, nps_score, sentiment 모두 없으면 텍스트 기반 감성 자동 분류 수행

#### 선택 — 있으면 해당 분석 활성화

| 컬럼 | 타입 | 설명 | 없을 때 영향 |
|------|------|------|-------------|
| `at` / `date` | datetime | 작성 일시 | 시간 추세 분석 제외 |
| `thumbsUpCount` | integer | 공감수/좋아요 수 | 공감 기반 우선순위 산출 불가, 건수 기반으로 대체 |
| `channel` | string | 수집 채널 (앱 리뷰, NPS, 이메일 등) | 채널별 분석 제외 |
| `userName` | string | 작성자 이름/ID | 동일 사용자 반복 분석 제외 |
| `replyContent` | text | 공식 답변 내용 | 답변률 분석 제외 |
| `topic` / `product_area` | string | 사전 분류된 토픽 | 키워드 기반 자동 분류 |
| `urgency` | string (high/medium/low) | 긴급도 | 긴급도 섹션 제외 |
| `plan` / `segment` | string | 고객 등급/플랜 | 세그먼트별 분석 제외 |
| `account_id` | string | B2B 계정 ID | 계정 가중 분석 제외 |
| `ticket_id` | string | CS 티켓 ID | CS 텍스트 패턴 적용 안 됨 |

### 1.4 데이터 소스 자동 감지

| 컬럼 조합 | 추정 소스 | 적용 패턴 |
|----------|----------|----------|
| `score` (1-5) + `content` | 앱 리뷰 | 토픽→감성 순서, 양극화 해석 |
| `nps_score` (0-10) + `content` | NPS 서술형 | 점수 구간(추천/중립/비추천)→토픽 순서 |
| `ticket_id` + `content` | CS 텍스트 | 진술 vs 근본원인 분리, 부정 편향 보정 |
| `account_id` + `content` | B2B 피드백 | 계정 매출 가중, RICE 우선순위 |
| 위 모두 없음 | 일반 텍스트 | 토픽→감성 기본 순서 |

### 1.5 최소 권장 건수
- **100건 이상**: 토픽 분류와 양극화 해석이 유의미
- **300건 이상**: 토픽별 감성 교차가 안정적
- **50건 미만**: 리포트 생성 가능하나 통계적 신뢰도 낮음 경고 표시

## 2. 설계 원칙

### 단일 골격
VOC 데이터는 앱 리뷰·CS 티켓·NPS 서술형·B2B 피드백·소셜 멘션 등 형태가 다양하지만, **리포트 골격은 동일**하다. 데이터 소스에 따라 슬롯에 들어가는 *지표 이름*과 *세부 콘텐츠*만 달라진다.

| 변하는 것 | 변하지 않는 것 |
|---|---|
| KPI의 구체 이름 (별점/NPS/CSAT) | KPI 슬롯의 역할 (볼륨, 만족도 대표값, 부정률, 분포 특이성) |
| 토픽 분류 결과 (가격/한도 vs 결제/환불) | "토픽 × 감성" 섹션이 존재한다는 사실 |
| 분포 해석 메시지 (양극화 vs Detractor 집중) | "분포 패턴을 해석한다"는 섹션의 목적 |
| 액션의 구체 내용 | 즉시/단기/중기 3구간으로 정리한다는 형식 |

### "필수 슬롯" 원칙
- 모든 섹션은 **항상 자리에 있다**. 데이터가 부족하면 채우는 콘텐츠가 줄어들 뿐, 슬롯 자체는 유지.
- 단, 데이터 자체가 0건이거나 의미 없는 경우(예: 긍정 비율 0%인데 긍정 분석)에는 **명시적 fallback 메시지**로 채움 ("이 데이터에서는 긍정 시그널이 발견되지 않았습니다").
- AI팀이 구현 시 **섹션을 동적으로 빼지 않는다** — 일관된 사용자 경험.

### 본 명세가 정의하는 것 / 정의하지 않는 것
| 정의함 | 정의하지 않음 |
|---|---|
| 7개 섹션의 순서·역할·슬롯 | 토픽 분류 알고리즘 |
| 각 슬롯이 받는 데이터의 *형태* | LLM 프롬프트 |
| 콘텐츠 규칙 (두괄식, VOC 톤) | 컴포넌트 렌더링 코드 |
| 좋은 예 / 나쁜 예 | 소스별 디테일 (→ skills/voc-analysis.md 참조) |

## 3. 리포트 골격 — 7개 섹션 (고정)

```
1. Executive Summary           — 전체 요약
2. 감성 스냅샷                  — 한눈에 보는 KPI 4개
3. 분포 패턴 해석               — "왜 이런 분포가 나오는가"
4. 토픽 × 감성                  — 어떤 주제에서 부정인가
5. 부정 심층                    — 상위 부정 토픽의 원인
6. 긍정 분석                    — 지켜야 할 가치
7. 액션 플랜                    — 즉시/단기/중기
```

이 순서는 **고정**. PM/Strategy Writer는 이 순서를 임의로 바꾸지 않는다.

---

## 4. 섹션별 슬롯 명세

각 섹션은 **슬롯(slot)** 단위로 정의됨. 슬롯 = "이 자리에 무엇이 들어가야 하는가"의 추상 설명. 데이터에 따라 슬롯을 채우는 구체 값만 달라짐.

---

### 섹션 1. Executive Summary
- **컴포넌트**: `ExecutiveSummary`
- **목적**: 리포트를 1분 안에 파악할 수 있게 핵심만 압축

| 슬롯 | 역할 | 데이터 부재 시 |
|---|---|---|
| `description` | **리포트 메타 설명** — 어떤 데이터를 무슨 축으로 분석한 어떤 종류의 리포트인지 (두괄식 미적용, 인사이트 X) | 분량 줄여 1줄로 |
| `topMetrics[0]` | **총 리뷰** (데이터 볼륨) | 항상 채워짐 |
| `topMetrics[1]` | **평균 별점** (만족도 대표값) | score 없으면 "감성 자동 분류" 표시 |
| | ※ 양극화율·부정비율 등 분석 인사이트는 상단 KPI가 아님 — 본문 섹션에서 다룬다 | |
| `keyFindings[0]` | 현황 진단 (전체 감성 한 줄) | 항상 채워짐 |
| `keyFindings[1]` | 분포 패턴 해석 (있을 때) / 또는 가장 큰 이상치 | 데이터에 따라 |
| `keyFindings[2]` | 우선순위 (임팩트 1-3위 토픽) | 토픽 추출이 되면 항상 |
| `keyFindings[3]` | 해결 방향 (action plan 요약) | 항상 |

- **콘텐츠 규칙**:
  - `description`은 **두괄식 미적용** — 리포트 메타 설명 (이게 어떤 데이터를 분석한 어떤 리포트인지)
  - `keyFindings`는 두괄식 + 4개가 순서대로 읽히면 하나의 이야기
  - `topMetrics`는 KPI 슬롯 §5 우선순위 적용

---

### 섹션 2. 감성 스냅샷
- **컴포넌트**: `MetricHighlight` (items 3개)
- **목적**: 전체 감성을 가장 강조해야 할 3개 지표로 표현

| 슬롯 | 역할 | 채우는 예시 |
|---|---|---|
| `items[0]` | 가장 두드러진 감성 지표 | 양극화 지수 / Detractor 집중도 / 부정률 |
| `items[1]` | 강도 지표 (얼마나 심한가) | 부정 비율 / NPS / CSAT |
| `items[2]` | 확산 지표 (얼마나 퍼져 있는가) | 공감 집중도 / 토픽 1위 비중 / SoV |

- **각 item 필드 규칙**:
  - `label`: KPI 카테고리 명 — 데이터에서 자동 도출
  - `value`: 실제 수치 — **두괄식 미적용** (이건 진짜 값)
  - `sub`: 산출 근거 1줄 (예: "1점 102건(20.4%) + 5점 306건(61.2%)")
  - `description`: 두괄식 인사이트 — 이 수치가 왜 중요한지

- **데이터 부재 시 fallback**: 슬롯을 비우지 말고 가장 가까운 대체 지표로 채움
  - 별점 없으면 → 감성 자동분류 결과
  - 공감수 없으면 → 토픽 비중
  - NPS 없으면 → CSAT or 평균 점수

---

### 섹션 3. 분포 패턴 해석
- **컴포넌트**: `ClusterCard` (items 2개)
- **목적**: "왜 이런 분포가 나오는가"를 해석. 분포가 드러나는 어떤 축이든 적용 가능.

| 슬롯 | 역할 | 채우는 예시 (소스별) |
|---|---|---|
| 카드 1 | 부정/문제 그룹의 행동 특성 | 1점 그룹 (앱 리뷰) / Detractor (NPS) / 에스컬레이션 티켓 (CS) |
| 카드 2 | 긍정/만족 그룹의 행동 특성 | 5점 그룹 / Promoter / 즉시 해결 티켓 |

- **각 카드 규칙**:
  - `badge`: 점수/구간 라벨 (예: "1점 그룹", "Detractor", "에스컬레이션 그룹")
  - `badgeColor`: 부정 그룹 `#f87171`, 긍정 그룹 `#7ccf00`
  - `title`: **군집 이름** (페르소나/그룹 명칭). 인사이트는 섹션 label에 위치. §7.5 참조
  - `description`: 프록시 지표(글자수·해결시간·재방문 등)로 행동 차이 해석

- **데이터 부재 시 fallback**: 분포 패턴이 미약하면 (예: 부정·긍정 비율이 비슷) **카드 1개로 축소** 또는 "유의미한 분포 차이가 발견되지 않음" 메시지로 단일 카드 표시

- **좋은 예 / 나쁜 예**:
  ```
  badge:       "1점 그룹"             ← 점수 라벨
  title:       "구체적 불만 기술자"     ← 군집 이름 (페르소나)
  description: "리뷰 평균 74자로 5점 대비 3.7배..."  ← 행동 특성 해석

  나쁜 예 (title에 인사이트 메시지):
  title: "불만은 길고 구체적이며 다른 사용자의 공감을 얻습니다"
  → 이건 섹션 label 또는 description으로 이동
  ```

### 섹션 4. 토픽 × 감성
- **컴포넌트**: `StackedBarChart`
- **목적**: 어떤 주제에서 부정이 집중되는지 한눈에. **VOC의 핵심 섹션**.

| 슬롯 | 역할 |
|---|---|
| `title` | "토픽별 감성 분포 ({정렬기준}순 정렬)" |
| `categories` | `["긍정", "중립", "부정"]` (고정) |
| `colors` | `["#7ccf00", "#e6e7e9", "#f87171"]` (고정) |
| `unit` | `"%"` (고정) |
| `items` | 토픽 행 — 표시 수 = `min(8, 토픽수)`, 정렬 = `impact` 내림차순 |

- **각 row label 규칙**: `"{토픽명} ({비중}%)"` — 비중을 라벨에 반드시 포함

- **데이터 부재 시 fallback**:
  - 토픽 수 < 3 → StackedBarChart 대신 InsightCard 1장으로 축소 (섹션 5에 흡수)
  - 그래도 슬롯은 유지 — "토픽 분류 결과 N개 — 단일 InsightCard로 표시"

- **섹션 라벨 규칙**: `"{1위토픽}·{2위토픽}·{3위토픽} — 비중×부정률 기준 개선 우선순위"`

---

### 섹션 5. 부정 심층
- **컴포넌트**: `InsightCard` (items 1-3개, 데이터에 따라 동적)
- **목적**: 상위 부정 토픽의 **근본 원인**을 고객 원문으로 진단

| 슬롯 | 역할 |
|---|---|
| 카드 1 | 임팩트 1위 부정 토픽 — 무엇이, 왜 |
| 카드 2 | 임팩트 2위 (있을 때) |
| 카드 3 | 임팩트 3위 (있을 때) |

- **각 카드 규칙**:
  - `badge`: 토픽명 (1-2단어)
  - `title`: **명사구 헤드라인** (1줄, ~20자, 동사 종결 금지). 수치 금지. §7.6 참조
  - `description`: 임팩트 순위 + 대표 verbatim 1-2개 + 공감수(있을 때)
  - `interpretation`: **근본 원인** (해결책 X. 인과 체인 형식)

- **콘텐츠 규칙**:
  - 카드 간 내용 겹침 금지 — 각 카드가 서로 다른 차원
  - 원문 인용은 description에서만, title에 금지
  - "분노/좌절/무력감" 같은 감정 판단 표현 금지

- **데이터 부재 시 fallback**: 부정 토픽이 0개면 "이 데이터에서는 유의미한 부정 토픽이 없습니다 (전체 부정률 {x}%)" 카드 1장으로 표시. 슬롯 유지.

- **좋은 예 / 나쁜 예**:
  ```
  나쁨 (수치 우선):       "임팩트 1위(7.0), 56건 중 부정 63%"
  나쁨 (줄글 어조):       "한도 예측 불가성이 결제 신뢰까지 훼손합니다"
  좋음 (명사구 헤드라인): "한도 예측 불가성이 결제 신뢰까지 훼손"
  ```

### 섹션 6. 긍정 분석
- **컴포넌트**: `InsightCard` (items 1-2개)
- **목적**: "고치면서 잃으면 안 되는 것"을 식별

| 슬롯 | 역할 |
|---|---|
| 카드 1 | 핵심 가치 — 긍정 테마 상위 묶음 |
| 카드 2 (조건부) | 이탈 신호 — 경쟁사 언급이 있을 때 |

- **각 카드 규칙**:
  - 카드 1 `title`: "지켜야 할 핵심 가치" 류의 메시지
  - 카드 2 `title`: "이탈 시그널" 류의 메시지
  - description에 verbatim 직접 인용

- **데이터 부재 시 fallback**:
  - 긍정 테마 0개 → 카드 1을 "긍정 시그널이 약합니다 (전체 긍정률 {x}%)"로 채움
  - 경쟁사 언급 0건 → 카드 2 슬롯은 비움 (1-카드 InsightCard로 렌더링)

---

### 섹션 7. 액션 플랜
- **컴포넌트**: `StrategyTable`
- **목적**: 부정 심층에서 도출된 원인을 즉시/단기/중기 액션으로 분리

| 슬롯 | 역할 | 행 수 |
|---|---|---|
| `immediate` | 즉시 가능한 핫픽스 | 1-2개 |
| `short` | 단기 (1-3개월) 메커니즘 개선 | 1-2개 |
| `mid` | 중기 (3-6개월) 시스템 재설계 | 1-2개 |

- **각 행 필드**:
  - `strategy`: 행동 지향 1줄 (예: "한도 리셋 시점 투명화")
  - `objective`: 어떤 불만/지표를 해소하는지 (수치 근거 포함)
  - `actionPlan`: 실행 방법 1-2줄
  - `expectedImpact`: 정량 목표 (예: "한도 관련 부정률 63%→25%")

- **콘텐츠 규칙**:
  - 모든 strategy는 섹션 5(부정 심층) 카드와 1:1 연결 (원인 → 해결책)
  - 마지막 mid 항목은 **종합 부정률 목표** (전체 KPI 개선)

## 5. 슬롯 채우기 로직 — 어떤 데이터가 들어와도 동일 골격

각 슬롯은 다음 우선순위로 채워진다. AI팀은 이 우선순위를 **공통 함수**로 구현.

### 만족도 대표값 슬롯 (Section 1 `topMetrics[1]`, Section 2 `items[1]`)
```
1순위: nps_score 컬럼 → "NPS X점"
2순위: score 컬럼 (1-5) → "평균 별점 X.X / 5.0"
3순위: csat_score 컬럼 → "CSAT X.X / 5.0"
4순위: sentiment 컬럼 → "긍정 비율 X%"
5순위: 텍스트 자동 감성 분류 결과 → "긍정 비율 X% (자동 분류)"
```

### 분포 특이성 지표 슬롯 (Section 1 `topMetrics[2]`, Section 2 `items[0]`)
```
1순위: polarization.detected → "양극화 지수 X%"
2순위: NPS Detractor 비율 → "Detractor 집중도 X%"
3순위: 부정 토픽 1위 비중 → "{토픽명} 집중도 X%"
4순위: 부정률 자체 → "전체 부정 비율 X%" (이때 Section 1 topMetrics[3]과 중복 → topMetrics[2]는 "—")
```

### 분포 패턴 해석 슬롯 (Section 3)
```
1순위: 양극화 (1+5점 70%+) → 1점 그룹 vs 5점 그룹
2순위: NPS 양극단 (Promoter+Detractor 70%+) → Detractor vs Promoter
3순위: 해결시간 양극단 → 즉시 해결 그룹 vs 에스컬레이션 그룹 (CS)
4순위: 글자수 양극단 → 짧은 반응 그룹 vs 긴 반응 그룹 (소스 무관)
5순위: 모두 해당 없음 → 단일 카드 fallback ("유의미한 분포 차이가 발견되지 않음")
```

### 공감/우선순위 가중 슬롯 (모든 verbatim 정렬)
```
1순위: thumbsUpCount (앱 리뷰)
2순위: ticket_priority (CS)
3순위: account_revenue (B2B)
4순위: 글자수 (긴 리뷰가 더 구체적)
5순위: 등록 순서 (random fallback)
```

## 7. 콘텐츠 규칙 (VOC 특수)

> **글로벌 콘텐츠 규칙** (두괄식, 명사구 헤드라인, ClusterCard.title=군집 이름, ES.description=메타 설명, InsightCard 세 필드 역할 분리, 섹션 label=두괄식 헤드라인) → [sample-generator.md](../../sample-generator.md) §0~§0-4 참조
>
> 본 섹션은 **VOC 도메인에만 적용되는 톤·인용 규칙**만 정의.

### 7.1 톤 — 감정 판단 표현 금지
- "분노/좌절/무력감" 등 감정 판단 표현 금지 → "불만/부정/이탈 위험" 등 중립적 표현
- 이유: VOC는 고객 텍스트 기반이라 LLM이 감정 명사를 과도하게 사용하는 경향이 있음

### 7.2 고객 원문 인용 위치
- 고객 verbatim은 **InsightCard `description`에서만** 인용
- ClusterCard, InsightCard `title`, 섹션 label, ES keyFindings 어디에도 따옴표 인용 금지
- 인용 시 출처 표기: `'원문'(공감 N)` 형식

### 7.3 양극화 해석 필수
- 1+5점 합산 70%+ 양극화가 감지되면 §4 섹션 3 (분포 패턴 해석)에서 반드시 해석
- "1점과 5점에 73% 집중"은 발견일 뿐 — 해석 X. 프록시 지표(글자수·공감수·짧은 리뷰 비율 등)로 행동 차이 진단 필수

## 8. AI팀 검수 체크리스트

### 6.1 골격 일관성
- [ ] 어떤 입력 데이터든 7개 섹션이 모두 렌더링되는가
- [ ] 데이터 부재 시 슬롯을 비우지 않고 fallback 메시지로 채우는가
- [ ] 섹션 순서가 1→7 고정인가

### 6.2 슬롯 채우기
- [ ] 만족도 대표값 우선순위(§5)가 정확히 적용되는가
- [ ] 분포 특이성 지표 우선순위가 정확히 적용되는가
- [ ] 분포 패턴 해석 fallback (단일 카드)이 작동하는가
- [ ] 공감/우선순위 가중이 데이터 소스에 따라 적절히 선택되는가

### 6.3 콘텐츠 규칙
- [ ] InsightCard `title` / ClusterCard `title` / 모든 description이 두괄식인가
- [ ] InsightCard `title`에 고객 원문 인용이 없는가
- [ ] 섹션 라벨에 수치가 포함되어 있는가 (ES 제외)

### 6.4 데이터 정합성
- [ ] StackedBarChart의 각 행 values 합이 100인가
- [ ] StrategyTable의 strategy가 부정 심층 카드와 1:1 매칭되는가
- [ ] expectedImpact의 출발 수치가 실제 preprocessor 출력과 일치하는가

## 9. 미해결 / 결정 필요

본 명세에 명시 못 한 항목 — AI팀과 협의 필요:

- **NPS·CS·B2B 데이터의 preprocessor 확장**: 현재 `voc-preprocessor.ts`는 앱 리뷰(score 컬럼) 위주. NPS Detractor 비율, CS 에스컬레이션율, B2B 계정 가중 등을 preprocessor에 추가해야 §5의 슬롯 채우기가 가능.
- **시계열 섹션 추가 여부**: `dateRange`가 있고 분석 기간이 4주 이상일 때 "시계열 추세" 섹션을 8번째 슬롯으로 추가할지 (현재 명세는 7개 고정).
- **비율 합 보정**: `sentimentBreakdown`이 반올림으로 100이 안 될 때, 렌더링 직전 가장 큰 값에 ±1% 보정 룰 명문화 필요.
- **Verbatim 익명화**: 고객 원문에 개인정보(이름·전화번호 등) 포함될 가능성. preprocessor에서 자동 마스킹 필요한지 결정.
