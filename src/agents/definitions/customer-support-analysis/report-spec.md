# Customer Support Analysis 리포트 명세

> AI팀 개발 핸드오프용. **어떤 CS 티켓 데이터가 들어와도 동일한 7개 슬롯**으로 구성되는 범용 리포트.
>
> 본 문서는 입력 데이터 정의(§1)부터 출력 리포트 골격(§3~§9)까지 단일 진입점으로 통합되어 있습니다.
>
> 관련 파일:
> - 도메인 지식 → [skills/customer-support.md](../../skills/customer-support.md), [skills/agent-performance.md](../../skills/agent-performance.md)
> - 컴포넌트 데이터 스키마 → [src/lib/constants.ts](../../../lib/constants.ts) `COMPONENT_DATA_SCHEMAS`
> - 콘텐츠 규칙 (두괄식 등) → [sample-generator.md](../../sample-generator.md), [strategy-writer.md](../../strategy-writer.md)
> - 자매 명세 (참조용) → [voc-analysis/report-spec.md](../voc-analysis/report-spec.md)

## 1. 입력 데이터

### 1.1 입력 형식
CSV 또는 JSON. 1행 = 1건의 지원 티켓.

### 1.2 컬럼 정의

#### 필수 (없으면 리포트 생성 불가)

| 컬럼 | 타입 | 설명 | 예시 |
|------|------|------|------|
| `ticket_id` | string | 티켓 고유 ID | TK-00883 |
| `category` | string | 문의 카테고리 | 결제/환불, 기능 오류, 계정/인증 |
| `status` | string | 처리 상태 | resolved, open, pending |
| `created_at` | datetime | 티켓 생성 일시 | 2026-01-15T09:23:00 |
| `resolved_at` | datetime | 해결 완료 일시 (status=open이면 null) | 2026-01-16T11:45:00 |
| `csat_score` 또는 `nps_score` | integer | 만족도 지표 (둘 중 하나 이상 필수) | csat_score: 1-5 / nps_score: 0-10 |

> **시간 컬럼**(`created_at`+`resolved_at`): 평균 해결시간 산출과 시간대/시계열 분석의 기본.
>
> **만족도 지표**(`csat_score` 또는 `nps_score` 중 하나): CS 리포트의 핵심 가치는 만족도 진단 — 둘 다 없으면 운영 통계로 전락하므로 최소 하나 필수. 우선순위는 §5 슬롯 채우기 로직 참조.

#### 선택 — 있으면 해당 분석 활성화

| 컬럼 | 타입 | 설명 | 없을 때 영향 |
|------|------|------|-------------|
| `priority` | string | 우선순위 (Critical/High/Normal/Low) | 우선순위별 SLA 분석 제외 |
| `channel` | string | 접수 채널 (채팅, 이메일, 전화, 웹폼) | 채널 편차 분석 제외 |
| `agent_id` | string | 담당 상담원 ID | 상담원 성과 비교 제외 |
| `assigned_at` | datetime | 담당자 배정 일시 | 병목 분해를 "대기 vs 처리"로 못 함 → 시간대 분해로 대체 |
| `escalated` | boolean | 에스컬레이션 여부 | 에스컬레이션율 산출 불가 → 카테고리 1위 비중으로 대체 |
| `csat_score` (nps만 있을 때) | integer (1-5) | CSAT 보조 지표 | NPS만으로 만족도 진단 |
| `nps_score` (csat만 있을 때) | integer (0-10) | NPS 보조 지표 | CSAT만으로 만족도 진단, NPS 분포 섹션 제외 |

### 1.3 컬럼 → 섹션 강화 매핑

(§4 섹션 카탈로그의 "데이터 의존" 슬롯과 동일한 정보를 컬럼 관점에서 정리)

| 리포트 섹션 | 필수 컬럼만으로 가능한 것 | 선택 컬럼이 있으면 강화 |
|------------|----------|---------------------|
| Executive Summary | KPI 4개 (만족도·속도·이상성 모두) | escalated → 이상성 슬롯 정량화 |
| 건강 KPI (MetricHighlight) | 만족도(CSAT/NPS), 평균 해결시간, 카테고리 집중도 | escalated → 에스컬레이션율로 이상성 슬롯 강화 |
| 카테고리 단면 (DataTable) | 건수, 평균 해결시간, 만족도(CSAT/NPS) | escalated → 에스컬레이션율 행 |
| 병목 분해 (StackedBarChart) | 시간대 분해 (영업/야간) | assigned_at → 대기 vs 처리 분해 |
| 처리 편차 (DataTable/StackedBar) | 만족도 분포 (NPS 3구간 또는 CSAT 추세) | agent_id → 상담원 편차, channel → 채널 편차 |
| 근본 원인 (InsightCard) | 만족도×카테고리·시간 교차 진단 (3카드) | escalated → 시스템 차원 정량화, agent_id → 역량 차원 |
| 액션 플랜 (StrategyTable) | 만족도·카테고리·시간 기반 액션 | 전부 |

### 1.4 섹션 자동 강화 규칙

| 추가 컬럼 | 강화되는 슬롯 |
|------|--------------|
| `+ assigned_at` | 병목 분해를 "시간대" → "대기 vs 처리"로 정밀화 |
| `+ agent_id` | 처리 편차의 1순위 = 상담원 편차 (DataTable) |
| `+ channel` | 처리 편차의 채널 편차 (DataTable) |
| `+ escalated` | KPI 이상성 슬롯 = 에스컬레이션율 (1순위), 근본 원인 "프로세스" 카드 정량화 |
| `+ csat_score` (nps만 있을 때) / `+ nps_score` (csat만 있을 때) | 만족도 진단의 두 축(직전 경험 + 추천 의향) 동시 분석 |

### 1.5 최소 권장 건수
- **200건 이상**: 카테고리별 교차 분석이 유의미
- **500건 이상**: 상담원 성과 비교가 안정적 (상담원당 30건+)
- **100건 미만**: 리포트 생성 가능하나 통계적 신뢰도 낮음 경고 표시

## 2. 설계 원칙

### 단일 골격
CS 티켓 데이터는 SaaS·이커머스·게임·금융 등 도메인에 따라 형태가 다양하지만(필수 컬럼이 `category`/`status`만 있고 나머지는 선택), **리포트 골격은 동일**하다. 슬롯에 들어가는 *지표 이름*과 *세부 콘텐츠*만 달라진다.

| 변하는 것 | 변하지 않는 것 |
|---|---|
| 만족도 지표 (CSAT vs NPS vs 자체 점수) | "건강 KPI 슬롯의 역할 (만족도/속도/이상성) |
| 분해 차원 (대기/처리, 단계별, 카테고리별) | "병목 분해" 섹션의 목적 |
| 편차 비교 축 (담당자/채널/시간/팀 중 가용한 것) | "처리 편차" 섹션의 존재 |
| 카테고리 종류 (결제/기능/계정 vs 배송/반품/상품) | 카테고리별 단면을 보여준다는 사실 |

### "필수 슬롯" 원칙 (VOC와 동일)
- 7개 섹션은 **항상 자리에 있다**. 데이터 부족 시 채우는 콘텐츠가 줄어들 뿐, 슬롯 자체는 유지.
- 데이터 자체가 0건이거나 의미 없으면 **명시적 fallback 메시지**로 채움 ("이 데이터에는 담당자 정보가 없어 편차 분석을 생략합니다").
- AI팀이 구현 시 **섹션을 동적으로 빼지 않는다** — 일관된 사용자 경험.

### 본 명세가 정의하는 것 / 정의하지 않는 것
| 정의함 | 정의하지 않음 |
|---|---|
| 7개 섹션의 순서·역할·슬롯 | SLA 임계치 / 카테고리 분류 알고리즘 |
| 각 슬롯이 받는 데이터의 *형태* | LLM 프롬프트 |
| 콘텐츠 규칙 (두괄식, 명사구 헤드라인) | 컴포넌트 렌더링 코드 |
| 좋은 예 / 나쁜 예 | 도메인별 디테일 (→ skills/customer-support.md 참조) |

## 3. 리포트 골격 — 7개 섹션 (고정)

```
1. Executive Summary           — 전체 요약
2. 건강 KPI                     — 한눈에 보는 KPI 3개 (만족도/속도/이상성)
3. 카테고리 단면                — 카테고리별 핵심 지표 한눈에 비교
4. 병목 분해                    — 어디서 시간/단계가 막히는가
5. 처리 편차                    — 누가/어디서/언제 다른가 (담당자/채널/시간 중 가용)
6. 근본 원인                    — 운영·시스템·역량 3차원 진단
7. 액션 플랜                    — 즉시/단기/중기
```

이 순서는 **고정**. PM/Strategy Writer는 이 순서를 임의로 바꾸지 않는다.

## 4. 섹션별 슬롯 명세

각 섹션은 **슬롯(slot)** 단위로 정의됨. 슬롯 = "이 자리에 무엇이 들어가야 하는가"의 추상 설명.

---

### 섹션 1. Executive Summary
- **컴포넌트**: `ExecutiveSummary`
- **목적**: 리포트를 1분 안에 파악할 수 있게 핵심만 압축

| 슬롯 | 역할 | 데이터 부재 시 |
|---|---|---|
| `description` | **리포트 메타 설명** — 어떤 데이터를 무슨 축으로 분석한 어떤 종류의 리포트인지 (두괄식 미적용, 인사이트 X) | 분량 줄여 1줄로 |
| `topMetrics[0]` | 데이터 볼륨 (총 티켓) | 항상 채워짐 |
| `topMetrics[1]` | 만족도 대표값 (CSAT·NPS·자체점수 중) | "—" 또는 "데이터 없음" |
| `topMetrics[2]` | 속도/SLA 대표값 (평균 해결시간·SLA 위반율 등) | created_at+resolved_at 부재 시 "—" |
| `topMetrics[3]` | 이상성 지표 (에스컬레이션율·재오픈율·대기율 등) | escalated 부재 시 "—" |
| `keyFindings[0]` | 현황 진단 (벤치마크 대비 위치 한 줄) | 항상 |
| `keyFindings[1]` | 가장 큰 문제 카테고리/구간 | 카테고리 분포 있으면 항상 |
| `keyFindings[2]` | 병목 원인 (왜 느린가/막히는가) | 병목 분해 가능할 때 |
| `keyFindings[3]` | 해결 방향 (action plan 요약) | 항상 |

- **콘텐츠 규칙**:
  - `description`은 **두괄식 미적용** — 리포트 메타 설명
  - `keyFindings`는 두괄식 + 4개가 순서대로 읽히면 하나의 이야기
  - `topMetrics`는 KPI 슬롯 §5 우선순위 적용

---

### 섹션 2. 건강 KPI
- **컴포넌트**: `MetricHighlight` (items 3개)
- **목적**: 운영 건강을 가장 강조할 3개 지표로 표현

| 슬롯 | 역할 | 채우는 예시 |
|---|---|---|
| `items[0]` | 만족도 대표값 + 벤치마크 격차 | "CSAT 3.70 (업계 4.0+)" / "NPS -22.4 (양호 +30~50)" |
| `items[1]` | 속도/SLA 대표값 + 벤치마크 | "평균 해결시간 27.4h" / "SLA 준수율 73%" |
| `items[2]` | 이상성 지표 (가장 두드러진 것) | "에스컬레이션율 30.9%" / "재오픈율 12%" / "1차 해결률 64%" |

- **각 item 필드 규칙**:
  - `label`: KPI 카테고리 명
  - `value`: 실제 수치 — **두괄식 미적용** (이건 진짜 값)
  - `sub`: 벤치마크 또는 비교 기준 1줄 (예: "업계 평균 4.0~4.2 대비 하회")
  - `description`: 두괄식 인사이트 — 이 수치가 왜 중요한지, 추세, 격차의 의미

- **데이터 부재 시 fallback**:
  - CSAT/NPS 모두 없으면 → "고객 만족도 데이터 없음 (CSAT/NPS 컬럼 부재)" 카드로 대체
  - 시간 데이터 없으면 → 카테고리 1위 비중 등 대체 지표

---

### 섹션 3. 카테고리 단면
- **컴포넌트**: `DataTable`
- **목적**: 카테고리별 핵심 지표를 한 표로 비교 — "어디서 일이 가장 많이/심각하게 일어나는가"

| 슬롯 | 역할 |
|---|---|
| `columns` | 카테고리 목록 (건수 내림차순, 최대 5-7개) |
| `rows` | 핵심 지표 (가용한 것만 행으로 추가) |
| `showDots` | `false` (기본). 클러스터 비교 아님 |

- **행 구성 우선순위** (있는 컬럼만 행으로 추가):
  1. **건수** — 항상 첫 행
  2. **에스컬레이션율** — escalated 컬럼 있을 때
  3. **평균 해결시간** — created_at + resolved_at 있을 때
  4. **CSAT** — csat_score 있을 때
  5. **NPS** — nps_score 있을 때
  6. **재오픈율 / 1차 해결률** — 가용 시

- **데이터 부재 시 fallback**:
  - category 컬럼 자체가 1종류면 → 채널·우선순위·시간대 등 다른 차원으로 대체
  - 모두 부재 → "카테고리 분류 데이터 없음" 메시지 1줄로 대체 (슬롯 유지)

- **섹션 라벨 규칙**: 가장 심각한 카테고리 1-2개 + 핵심 지표 = 헤드라인
  - 예: "기능 오류·결제/환불 두 카테고리가 SLA 위반의 핵심 — 에스컬레이션율 61.8%·38.1%"

---

### 섹션 4. 병목 분해
- **컴포넌트**: `StackedBarChart`
- **목적**: "왜 느린가" 또는 "왜 막히는가"를 분해 — 데이터에 따라 분해 축이 다름

| 슬롯 | 역할 | 분해 축 (우선순위) |
|---|---|---|
| `title` | "{축} 분해 ({분해 단위})" | 데이터에서 도출 |
| `categories` | 분해 구성요소 2-4개 | 아래 우선순위 |
| `items` | 카테고리/그룹별 행 (DataTable과 같은 정렬) | min(5, 그룹수) |

- **분해 축 우선순위**:
  1. **해결시간 분해** (대기 vs 처리) — created_at+assigned_at+resolved_at 있을 때
  2. **단계 분해** (L1/L2/Eng) — escalation_level 있을 때
  3. **상태 분해** (resolved/open/pending) — status 있을 때 (fallback)
  4. **시간대 분해** (영업/야간) — created_at만 있을 때

- **차트 파라미터**:
  - `colors`: 부정 단계는 `#f87171`, 중립은 `#e6e7e9`, 긍정은 `#7ccf00`
  - `unit`: 시간 분해면 `"h"`, 비율 분해면 `"%"`, 건수면 `"건"`

- **데이터 부재 시 fallback**: 분해 축이 하나도 안 잡히면 → InsightCard 1장으로 축소 ("이 데이터로는 병목 분해가 불가합니다")

- **섹션 라벨 규칙**: "{1위 그룹}의 {지표}가 핵심 — {분해 결과 한 줄}"
  - 예: "기능 오류 50.2시간 중 배정 대기가 22.1h — 카테고리 간 7배 격차의 내부"

---

### 섹션 5. 처리 편차
- **컴포넌트**: `DataTable` (또는 `StackedBarChart` — 분포 비교일 때)
- **목적**: 같은 일도 누가/어디서/언제 처리하느냐에 따른 편차 식별

| 슬롯 | 역할 | 편차 축 (우선순위) |
|---|---|---|
| 비교 그룹 | 상위 N · 하위 N · 평균 비교 | 아래 우선순위 |

- **편차 축 우선순위**:
  1. **담당자 편차** (agent_id 있을 때) — 상위 5명 vs 하위 5명 vs 전체
  2. **채널 편차** (channel 있을 때) — 채팅·이메일·전화·웹폼 비교
  3. **만족도 분포** (csat_score 또는 nps_score 있을 때) — 카테고리별 추천/중립/비추천 분포 (NPS의 경우)
  4. **시간 편차** (created_at 있을 때) — 시간대별/요일별 편차
  5. **모두 부재** → "편차 분석 가능한 차원이 없습니다" + 단순 평균만 1줄

- **컴포넌트 선택 규칙**:
  - 담당자/채널 편차 → `DataTable` (행: 지표, 열: 그룹)
  - 만족도 분포 → `StackedBarChart` (categories: 추천자/중립/비추천자)
  - 시간 편차 → `TrendLineChart` (있을 때)

- **섹션 라벨 규칙**: 편차의 크기 + 의미
  - 예 (담당자): "상담원 에스컬레이션율 편차 5.7%p — 교육으로 줄일 수 있는 여지"
  - 예 (NPS 분포): "기능 오류 비추천자 72.1%가 전체 NPS를 끌어내림"

---

### 섹션 6. 근본 원인
- **컴포넌트**: `InsightCard` (items **정확히 3개**)
- **목적**: 위 섹션들에서 본 현상의 **구조적 원인**을 3개 차원으로 진단

| 슬롯 | 차원 | 채우는 예시 |
|---|---|---|
| 카드 1 | **프로세스** — 흐름·배정·SLA 운영 | "수동 배정으로 대기 발생" |
| 카드 2 | **시스템** — 도구·자동화·셀프서비스 | "셀프서비스 부재로 단순 문의가 큐 점유" |
| 카드 3 | **역량** — 상담원 교육·일관성 | "상담원 편차로 해결 패턴 미표준화" |

- **각 카드 규칙**:
  - `badge`: 차원명 ("프로세스" / "시스템" / "역량")
  - `title`: **명사구 헤드라인** (1줄, ~20자, 동사 종결 금지)
  - `description`: 근거 — 수치, 카테고리, 비교
  - `interpretation`: **인과 체인** (원인 → 결과 → 영향). title/description 재진술 금지

- **데이터 부재 시 fallback**:
  - agent_id 부재 → 카드 3 (역량)을 "상담원 데이터 없음 — 역량 차원 진단 불가"로 대체
  - 카드 슬롯 3개는 유지

- **콘텐츠 규칙**:
  - 카드 간 내용 겹침 금지 — 3개 카드가 서로 다른 차원
  - 원문/티켓 인용은 description에서만, title에 금지

---

### 섹션 7. 액션 플랜
- **컴포넌트**: `StrategyTable`
- **목적**: 근본 원인 3차원에 대응하는 즉시/단기/중기 액션

| 슬롯 | 역할 | 행 수 |
|---|---|---|
| `immediate` | 즉시 가능한 운영 핫픽스 | 1-2개 |
| `short` | 단기 (1-3개월) 메커니즘 개선 | 1-2개 |
| `mid` | 중기 (3-6개월) 시스템 재설계 | 1-2개 |

- **각 행 필드**:
  - `strategy`: 행동 지향 1줄 (예: "기능 오류 자동 배정 도입")
  - `objective`: 어떤 지표를 어떻게 개선하는지 (수치 근거 포함)
  - `actionPlan`: 실행 방법 1-2줄
  - `expectedImpact`: 정량 목표 (예: "기능 오류 해결시간 50h→30h")

- **콘텐츠 규칙**:
  - 모든 strategy는 섹션 6(근본 원인) 카드와 1:1로 연결 (원인 → 해결책)
  - 마지막 mid 항목은 **종합 KPI 목표** (CSAT 또는 NPS 개선 등)

## 5. 슬롯 채우기 로직 — 어떤 데이터가 들어와도 동일 골격

각 슬롯은 다음 우선순위로 채워진다. AI팀은 이 우선순위를 **공통 함수**로 구현.

### 만족도 대표값 슬롯 (Section 1 `topMetrics[1]`, Section 2 `items[0]`)
```
1순위: csat_score 컬럼 → "CSAT X.X / 5.0"
2순위: nps_score 컬럼 → "NPS X점"
3순위: 자체 만족도 컬럼 (5점 척도) → "만족도 X.X / 5.0"
4순위: 모두 부재 → "—" 또는 "데이터 없음 (CSAT/NPS 컬럼 부재)"
```

### 속도/SLA 대표값 슬롯 (Section 1 `topMetrics[2]`, Section 2 `items[1]`)
```
1순위: created_at + resolved_at → "평균 해결시간 X.Xh"
2순위: SLA 컬럼 (있을 때) → "SLA 준수율 X%"
3순위: created_at + first_response_at → "평균 첫 응답 X분"
4순위: 모두 부재 → "—" + "시간 데이터 부재"
```

### 이상성 지표 슬롯 (Section 1 `topMetrics[3]`, Section 2 `items[2]`)
```
1순위: escalated 컬럼 → "에스컬레이션율 X%"
2순위: status에 reopen 값 있음 → "재오픈율 X%"
3순위: 1차 해결 가능 (escalated=false 비율) → "1차 해결률 X%"
4순위: 카테고리 1위 비중 → "{카테고리} 집중도 X%"
```

### 카테고리 단면 행 슬롯 (Section 3 `rows`)
```
순서: 건수 → (있을 때) 에스컬레이션율 → 해결시간 → CSAT → NPS → 재오픈율
표시 행 수: min(5, 가용 지표 수)
```

### 병목 분해 축 슬롯 (Section 4)
```
1순위: 해결시간 분해 (created_at+assigned_at+resolved_at)
2순위: 단계 분해 (escalation_level)
3순위: 상태 분해 (status)
4순위: 시간대 분해 (created_at만)
5순위: 모두 부재 → InsightCard fallback
```

### 처리 편차 축 슬롯 (Section 5)
```
1순위: 담당자 편차 (agent_id) — DataTable
2순위: 만족도 분포 (csat_score 또는 nps_score) — StackedBarChart
3순위: 채널 편차 (channel) — DataTable
4순위: 시간 편차 (created_at) — TrendLineChart
5순위: 모두 부재 → 단순 평균 1줄
```

## 6. 컴포넌트 데이터 스키마 (참조)

상세는 [src/lib/constants.ts](../../../lib/constants.ts) `COMPONENT_DATA_SCHEMAS`. 본 명세에 등장하는 컴포넌트만 요약:

```ts
ExecutiveSummary: {
  description: string,
  topMetrics: [{ label, value }],          // 4개
  keyFindings: [string]                     // 4개
}

MetricHighlight: {
  items: [{ label, value, sub?, description }]  // 1-3개
}

DataTable: {
  columns: [{ label }],
  rows: [{ metric: "행 라벨", values: ["string|number"] }],
  showDots: false                           // CS는 항상 false
}

StackedBarChart: {
  title?, categories: [string], colors?: [string],
  items: [{ label, values: [number] }],
  unit?: "h" | "%" | "건"
}

InsightCard: {
  items: [{ badge?, title, description, interpretation? }]  // 정확히 3개 (CS의 경우)
}

StrategyTable: {
  immediate: [{ strategy, objective, actionPlan, expectedImpact }],
  short:     [{ ... 같은 형태 ... }],
  mid:       [{ ... 같은 형태 ... }]
}
```

## 7. 콘텐츠 규칙 (CS 특수)

> **글로벌 콘텐츠 규칙** (두괄식, 명사구 헤드라인, ClusterCard.title=군집 이름, ES.description=메타 설명, InsightCard 세 필드 역할 분리, 섹션 label=두괄식 헤드라인) → [sample-generator.md](../../sample-generator.md) §0~§0-4 참조
>
> 본 섹션은 **CS 도메인에만 적용되는 톤·인용·차원 규칙**만 정의.

### 7.1 톤 — 평가/비난 표현 금지
- "비효율적", "무능한", "느린" 같은 평가 표현 자제 → "개선 여지", "벤치마크 미달", "처리 지연"
- **상담원 개인 비난 표현 금지** — "하위 5명"은 OK, "성과가 나쁜 상담원" X (이유: SynTitan 제품 성격상 상담원을 코칭 대상으로 보이게)
- 이유: CS 리포트는 운영 개선 의사결정용 — 평가 표현은 조직 갈등 유발

### 7.2 티켓 인용 위치
- 구체적 티켓 ID 또는 본문 인용 시 **InsightCard `description`에서만**
- title, 섹션 label, ES keyFindings에 인용 금지

### 7.3 InsightCard 3카드 차원 강제
- §4 섹션 6 (근본 원인) InsightCard는 **반드시 프로세스/시스템/역량 3차원**으로 카드 분리
- 카드 간 차원이 겹치면 안 됨 — "프로세스" 카드에서 "교육" 언급 금지 (그건 "역량" 카드 영역)

### 7.4 컴포넌트 배치 (CS 특수)
- DataTable이 연속해서 2개 오는 건 OK — §4 섹션 3(카테고리 단면) → 섹션 5(처리 편차)에서 자연스러운 흐름
- (§7.4 외 일반 배치 규칙은 sample-generator.md 참조)

## 8. AI팀 검수 체크리스트

### 6.1 골격 일관성
- [ ] 어떤 입력 데이터든 7개 섹션이 모두 렌더링되는가
- [ ] 데이터 부재 시 슬롯을 비우지 않고 fallback 메시지로 채우는가
- [ ] 섹션 순서가 1→7 고정인가

### 6.2 슬롯 채우기
- [ ] 만족도 대표값 우선순위(§5)가 정확히 적용되는가
- [ ] 속도/SLA 대표값 우선순위가 정확히 적용되는가
- [ ] 병목 분해 축이 데이터에 따라 적절히 선택되는가
- [ ] 처리 편차 축 우선순위(담당자 > 만족도 > 채널 > 시간)가 적용되는가

### 6.3 콘텐츠 규칙
- [ ] InsightCard.title이 두괄식 + 명사구인가
- [ ] interpretation이 인과 체인 형식인가 (원인→결과→영향)
- [ ] 섹션 라벨에 수치가 포함되어 있는가 (ES 제외)
- [ ] 상담원 개인 비난 표현 없는가

### 6.4 데이터 정합성
- [ ] DataTable 각 열의 카테고리가 일관되게 정렬되는가 (건수 내림차순)
- [ ] StackedBarChart 각 행의 values 합이 unit에 맞는가 (h: 시간 합, %: 100)
- [ ] StrategyTable의 strategy가 근본 원인 카드와 1:1 매칭되는가

## 9. 미해결 / 결정 필요

본 명세에 명시 못 한 항목 — AI팀과 협의 필요:

- **시간 분해 데이터 추출**: `created_at + assigned_at + resolved_at` 3개로 대기/처리를 분해해야 하는데, `assigned_at`이 표준 컬럼이 아님. 추정 룰 또는 컬럼 매핑 가이드 필요.
- **SLA 임계치 정의**: 카테고리별 SLA 시간을 어디서 가져올지 (도메인 기본값 vs 설정 파일).
- **Verbatim 익명화**: CS 티켓 본문에 개인정보(이름·이메일·결제정보) 포함 가능성. 마스킹 필요.
- **다국어 처리**: CS 티켓이 한국어/영어 혼재일 때 카테고리 분류·감성 처리 룰.
- **시계열 섹션 추가 여부**: 4주 이상 데이터일 때 8번째 슬롯으로 시계열 추세를 추가할지 (현재 명세는 7개 고정).
