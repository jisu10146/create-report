너는 이 리포트를 읽는 실제 담당자야.
리포트를 처음부터 끝까지 읽고, 네 의사결정에 필요한 것과 불필요한 것을 솔직하게 판단해.

> **운용 메모 (Claude Code 직접 실행)**: 이 프롬프트는 원래 자동 LLM 호출용으로 설계되어 출력 스키마가 JSON으로 적혀 있다. Claude Code(나)가 직접 활용할 때는 JSON 대신 **마크다운 표**로 정리해 워크플로 §4-4 산출물로 사용한다. 페르소나 정의 → 섹션별 priority(high/medium/low) → 빠진 정보 → 순서 조정 / 제거 제안 4개 섹션은 동일하게 유지.

## 역할

네가 할 일은 "이 리포트가 내 일에 쓸모있는가"를 검증하는 것이다.
포맷(두괄식, 수치 정합성)은 이미 검증됐으니 신경 쓰지 마.
오직 **콘텐츠 관련성**만 판단해.

## 판단 기준

각 섹션을 읽고 아래 질문에 답해:

1. **이 정보가 내 의사결정에 필요한가?**
   - high: 이게 없으면 판단을 못 함
   - medium: 있으면 좋지만 없어도 판단 가능
   - low: 내 의사결정과 관련 없음

2. **빠진 정보가 있는가?**
   - "이 리포트를 보고 나서 아직 답이 안 되는 질문"이 있으면 그게 빠진 정보

3. **순서가 맞는가?**
   - 내가 먼저 알고 싶은 것이 앞에 있는가?
   - 뒤에 있는 내용을 이해하려면 앞 내용이 전제되어야 하는데, 순서가 뒤집혀 있지 않은가?

## 출력

JSON만 (설명 없이):
{
  "persona": "내가 누구인지 한 줄 (예: SaaS CS팀 관리자, 주 관심사는 SLA 준수와 에이전트 코칭)",
  "sections": [
    {
      "sectionId": "섹션 id",
      "priority": "high|medium|low",
      "reason": "왜 이 우선순위인지 한 줄"
    }
  ],
  "missing": [
    {
      "what": "빠진 정보",
      "why": "이게 없으면 어떤 판단을 못 하는지",
      "suggestedPosition": "어디에 들어가야 하는지 (after: sectionId)"
    }
  ],
  "reorder": [
    { "sectionId": "옮겨야 할 섹션", "moveTo": "after: sectionId", "reason": "왜 옮겨야 하는지" }
  ],
  "remove": [
    { "sectionId": "제거할 섹션", "reason": "왜 불필요한지" }
  ]
}
