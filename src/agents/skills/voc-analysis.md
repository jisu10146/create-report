## Terminology (텍스트 분석 전용)

| term | definition |
|------|-----------|
| Sentiment Analysis | 감성 분석. 텍스트에서 긍정/중립/부정 감정 분류 |
| Topic Clustering | 주제 클러스터링. 유사 피드백을 자동 그룹화하여 핵심 주제 도출 |
| Verbatim | 고객이 직접 작성한 원문 텍스트. 정량 지표로 포착되지 않는 맥락과 감정 제공 |
| Theme | 피드백에서 반복적으로 등장하는 주제. Topic Clustering의 결과물 |
| Driver Analysis | 만족/불만족의 주요 동인을 통계적으로 식별 |
| Text Analytics | 비정형 텍스트에서 패턴, 감성, 주제를 추출하는 기술 |
| HDBSCAN | 밀도 기반 클러스터링 알고리즘. 유사 주제 통합에 활용 |
| Hallucination | AI가 사실이 아닌 정보를 사실처럼 생성하는 현상. AI 앱 리뷰의 핵심 불만 축 |
| Regression | 업데이트 후 기존 기능이 퇴보하는 현상. AI 앱에서 "모델이 멍청해졌다" 인식으로 나타남 |

## Report Reader Priorities

VOC 리포트를 보는 CX/프로덕트 관리자가 확인하는 순서.

| 순위 | 관리자가 보는 것 | 핵심 질문 | 적합 컴포넌트 |
|------|-----------------|----------|-------------|
| 1 | 전체 감성 스냅샷 | "고객이 전반적으로 어떻게 느끼는가?" | MetricHighlight |
| 2 | 별점 양극화 해석 | "왜 중간이 없는가? 어떤 사람이 1점/5점을 주는가?" | ClusterCard (페르소나 분리) |
| 3 | 토픽×감성 교차 | "어떤 주제에서 부정적인가? 고객이 어떤 단어로 표현하는가?" | StackedBarChart + Verbatim 인용 |
| 4 | 부정 토픽 심층 (고객 목소리) | "가장 심한 토픽에서 고객이 뭐라고 하는가?" | InsightCard (원문 인용 중심) |
| 5 | 긍정 토픽 (지켜야 할 것) | "고객이 좋아하는 건 뭔가? 잃으면 안 되는 것" | InsightCard 또는 BulletCard |
| 6 | 액션 플랜 | "무엇을 바꿔야 하는가?" | StrategyTable |

### VOC만의 구조 원칙 (CS/마케팅 리포트와 구분)
- **CS 리포트 틀을 복사하지 않는다** — VOC는 숫자가 아니라 텍스트가 핵심. "고객이 어떤 단어로 불만을 표현하는가"가 리포트의 가치
- **Verbatim을 테이블 1개에 가두지 않는다** — 모든 InsightCard·description에 고객 원문 인용을 포함
- **양극화가 발견되면 반드시 해석한다** — "1점과 5점에 집중"은 발견이지 분석이 아님. "어떤 사람이 1점을 주고, 어떤 사람이 5점을 주는가" 페르소나 분리 필수
- **긍정 분석도 필수** — 부정만 분석하면 "고쳐야 할 것"만 보임. "지켜야 할 것"도 있어야 균형

## VOC 데이터 소스별 분석 패턴

VOC는 앱 리뷰만이 아니다. 데이터 소스마다 구조·편향·분석법이 다르며, 소스를 먼저 파악한 뒤 해당 패턴을 적용해야 한다.

### 1. NPS/CSAT 서술형 응답
| 항목 | 특성 |
|------|------|
| 데이터 구조 | 짧음(1-2문장), 점수(0-10 또는 1-5)와 쌍으로 존재, 응답률 10-30% |
| 감성 분포 | 양극화 — 추천자/비추천자가 주로 작성, 중립자는 거의 쓰지 않음 |
| 분석 순서 | **점수 구간(추천/중립/비추천)으로 먼저 나눈 뒤 토픽 분류** — 앱 리뷰(토픽 먼저→감성)와 반대 |
| 핵심 지표 | 토픽별 NPS (예: "가격" 비추천자 vs "UX" 비추천자), 분기별 추이 |
| 함정 | 인상적인 인용문만 골라서 보고하는 cherry-picking. 빈도 기반 체계적 코딩 필수 |
| 함정 | 설문 피로(survey fatigue)로 후반 웨이브의 텍스트가 얕아짐 — 응답 길이 추세 모니터링 |

### 2. CS 티켓/채팅 텍스트 (VOC 관점)
| 항목 | 특성 |
|------|------|
| 데이터 구조 | 대화형(multi-turn), 에이전트 응답 혼재, 대량(월 수천~수만 건) |
| 감성 분포 | 강한 부정 편향 — 만족한 고객은 티켓을 열지 않음 |
| 분석 순서 | 고객의 "진술된 문제"와 "근본 원인" 분리 필수. "버그"로 분류된 32%가 실제로는 온보딩 마찰 |
| 핵심 지표 | 테마 빈도 × 해결 노력(또는 이탈 상관), 단순 건수 아님 |
| 함정 | CS팀의 카테고리를 VOC 카테고리로 착각. CS 분류는 워크플로 라우팅용("결제 문제"), VOC는 경험 기반("가격 페이지가 혼란스러움") |

### 3. B2B SaaS 제품 피드백 (인앱, 기능 요청)
| 항목 | 특성 |
|------|------|
| 데이터 구조 | 저볼륨·고신호. 15+ 분산 소스(Slack, QBR, 세일즈 노트, 인앱 위젯). 단위 = 개인이 아니라 계정 |
| 감성 분포 | 명시적 감성보다 "요청/제안" 형태가 많음 |
| 분석 순서 | **계정 매출/전략적 가치로 가중** — 엔터프라이즈 1건 > SMB 100건. RICE/Kano 모델로 우선순위 |
| 핵심 지표 | 기능 요청→채택율, 피드백 루프 클로저율(고객에게 결과를 알렸는가) |
| 함정 | 기능 요청을 근본 니즈와 혼동. "CSV 내보내기 추가해주세요" = 실제 니즈는 "재무팀과 데이터 공유" |
| 함정 | "회의실에서 가장 큰 목소리" 편향 — CRM 노트와 내부 추정으로 로드맵 결정 |

### 4. 소셜 미디어 / 커뮤니티 멘션
| 항목 | 특성 |
|------|------|
| 데이터 구조 | 비요청(unsolicited), 상시 스트림, 높은 노이즈 비율. 고객이 회사가 아닌 네트워크에게 말함 |
| 감성 분포 | 극단치 중심. 바이럴로 볼륨 급등 가능하나 실제 영향 범위와 무관할 수 있음 |
| 분석 순서 | 경쟁사 멘션이 자연스럽게 포함됨. 자동 감성 분석이 풍자/밈/문화 맥락에서 부정확 |
| 핵심 지표 | 점유율(Share of Voice) vs 경쟁사, 감성 추세(절대값 아님) |
| 함정 | 볼륨 = 중요도 착각. 바이럴 불만 스레드가 사용자 0.1%만 해당할 수 있음. 반대로 조용한 이탈(멘션 감소)은 감지 불가 |

### 5. 세일즈 콜 노트 / 이탈 인터뷰
| 항목 | 특성 |
|------|------|
| 데이터 구조 | 최고 품질 정성 데이터, 최저 볼륨. 의사결정 과정과 검토한 대안을 포함 — 다른 소스에 없는 맥락 |
| 감성 분포 | 이탈 고객은 회사에 직접 부정을 말하길 꺼림 — 제3자 인터뷰가 더 솔직한 결과 |
| 분석 순서 | 분석 단위 = 개별 문장이 아니라 내러티브 아크. 테마는 전략적(시장 적합, 경쟁 포지셔닝) |
| 핵심 지표 | Win/Loss 테마 일관성, 이탈 사유별 이탈 시점(time-to-churn) |
| 함정 | 이탈 사유가 사후 합리화일 수 있음. 실제 원인은 수개월간 누적된 마찰. 세일즈팀이 자기 내러티브에 맞게 필터링하여 기록 |

### VOC 소스 자동 감지 규칙
데이터 컬럼으로 소스를 추정하여 해당 패턴을 적용한다:
- `score` 1-5 + `content` → **앱 리뷰** 패턴
- `nps_score` 0-10 + `content` → **NPS 서술형** 패턴 (점수 구간→토픽 순서)
- `ticket_id` + `content` → **CS 텍스트** 패턴 (진술 vs 근본원인 분리)
- `account_id` + `content` → **B2B 피드백** 패턴 (계정 가중)
- `channel` = "twitter/reddit/community" → **소셜** 패턴 (노이즈 필터, SoV)
- 위 어느 것에도 해당하지 않으면 → 일반 텍스트 분석 (토픽→감성 순서)

## 앱 스토어 리뷰 분석 패턴

앱 리뷰는 VOC의 한 유형으로, 별점(score)과 텍스트(content)가 쌍으로 존재하는 구조다.

### 일반 앱 리뷰 분석 규칙
- 분석 순서: 토픽 분류 먼저 → 토픽 내 감성 분포 확인
- 양극화 임계치: 1점+5점 합산 70% 이상이면 양극화 해석 필수
- 프록시 지표: 리뷰 길이(글자수) = 사용 깊이 프록시
- 토픽 우선순위: 비중(건수 비율) × 부정률. 부정률만으로 정렬 금지
- 공감수(thumbsUp): 공감 높은 리뷰 = 다수가 동의하는 의견. 우선순위에 반영

## AI 챗봇/앱 리뷰 분석 패턴

AI 앱(ChatGPT, Gemini, Claude 등) 리뷰는 일반 앱과 다른 고유 패턴이 있다.

### AI 앱 고유 불만 토픽
| 토픽 | 일반 앱과의 차이 | 분석 시 주의점 |
|------|----------------|---------------|
| 응답 품질/할루시네이션 | 일반 앱에는 없는 카테고리. AI 앱의 핵심 가치 축 | "오답을 인정하지 않고 고집한다"가 단순 오류보다 더 강한 부정 유발 |
| 무료/유료 한도 | 일반 앱은 기능 제한이지만, AI 앱은 "대화 자체가 막힘" | 한도 리셋 시점 불투명이 핵심. 한도 자체보다 예측 불가성이 불만 |
| 모델 품질 퇴보 인식 | 일반 앱 = 버그 불만, AI 앱 = "모델이 멍청해졌다" | 실제 퇴보가 아닐 수 있음. 기대치 상승 + 경쟁사 경험이 원인일 수 있다 |
| 콘텐츠 검열/제한 | 일반 앱에는 드묾 | 소수지만 감정이 매우 강함. 부정률 100%에 가까움 |
| 경쟁사 직접 비교 | 일반 앱은 암시적, AI 앱은 "Gemini가 더 낫다"처럼 명시적 | 이탈 맥락 vs 비교 맥락 구분 필수. 이탈 권유는 강한 신호 |

### AI 앱 양극화 해석 프레임워크
- **5점의 허상**: AI 앱 5점 리뷰의 30-40%는 5자 이하 단순 반응 ("좋아요", "굿")
- **1점의 신호**: 1점 리뷰는 평균 글자수가 5점의 3-4배. 구체적 문제를 기술하는 진단서
- **프록시 지표**: 리뷰 길이(글자수)를 사용 깊이의 프록시로 활용 가능
  - 짧은 리뷰(~5자) = 습관적/감정적 반응
  - 긴 리뷰(50자+) = 구체적 경험 기반 피드백
- **공감수 역전**: 공감(thumbsUp)이 높은 리뷰의 대부분이 1점. 부정이 공감을 끌어내는 구조

### AI 앱 긍정 가치 축
| 가치 | 설명 | 위험 신호 |
|------|------|----------|
| 학습/정보 보조 | 즉시 정보 접근, 복잡한 질문 해결 | 할루시네이션이 이 가치를 직접 훼손 |
| 말동무/감정 지원 | AI만의 고유 가치. 대체 어려움 | 검열 강화 시 이 가치가 약화될 수 있음 |
| 편리성/범용성 | "뭐든 물어볼 수 있다"는 인식 | 한도 제한이 이 인식을 깨뜨림 |
| 이미지/창작 | 시각적 콘텐츠 생성 | 한도가 가장 먼저 부딪히는 기능 |

### AI 앱 경쟁사 언급 분석 규칙
- 경쟁사 언급은 절대 수가 적어도(~2%) 강한 이탈 신호
- 반드시 맥락을 구분할 것:
  - **이탈 맥락**: "~쓰세요", "~가 더 낫다" → 즉시 대응 필요
  - **비교 맥락**: "~도 써봤는데" → 병행 사용, 모니터링
  - **긍정 맥락**: "~보다 낫다" → 지켜야 할 차별점

## VoC 데이터 분석 프로세스

### 1단계: 핵심 주제 추출
- 전체 VoC 중 샘플(5~10%)을 먼저 검토하여 반복 주제 도출
- 주제는 5음절 이하, "형용사+명사" 형식 권장
- 2,000건이면 100~200건 샘플로 주제 추출

### 2단계: 개별 VoC ↔ 주제 매칭
- 모든 개별 VoC에 1~3개 주제 매칭 (최대 3개)
- 매칭 과정에서 새 주제 추가 가능
- HDBSCAN 등 밀도 기반 클러스터링으로 유사 주제 통합

### 3단계: 정량화 및 시각화
- 주제별 등장 빈도 집계 → 정량 데이터 변환
- 감성과 교차하여 "어떤 주제에서 부정이 집중되는가" 분석
- DonutChart(토픽 분포) + StackedBarChart(토픽×감성) 조합

### 분석의 주요 어려움과 대응
| 어려움 | 대응 |
|--------|------|
| 대량 데이터 | AI 자동 분류 (수만 건 수동 분류 불가) |
| 분류 일관성 | 클러스터링 알고리즘으로 유사 주제 통합 |
| 정보 손실 | **대표 VoC(Verbatim) 추출**로 보완 필수 |

### 리포트에서 정보 손실 보완
- 주제별 빈도만 보여주면 "왜"가 빠짐 → InsightCard에 대표 Verbatim 인용
- 정량 차트 옆에 반드시 정성 근거(Verbatim) 제시
- DataTable에 "고객 원문(요약)" 열 포함

## Common VoC Topics

| topic | 특성 | 분석 시 주의점 |
|-------|------|---------------|
| 제품 기능/UX | 가장 빈번, 긍정·부정 혼재 | 부정에서 "어떤 기능"인지 세부 분해 |
| 가격/비용 | 부정 비율 높음, 감정적 | 경쟁사 대비 언급 별도 추출 |
| 고객 지원 | 감성 양극화 | 에이전트별·채널별 분해 |
| 온보딩/학습 | 초기 고객 집중 | 가입 후 1-3개월에 집중 |
| 안정성/버그 | 강한 부정, 이탈 신호 | NPS 비추천자와 교차 시 이탈 예측 |
| 경쟁사 언급 | 비교 맥락, 이탈 리스크 | 어떤 맥락에서 언급하는지 Verbatim 확인 |
