## Escalation Tiers

| tier | from → to | when | 분석 시 체크포인트 |
|------|-----------|------|--------------------|
| L1 → L2 | 프론트라인 → 시니어 | 심화 조사, 전문 지식 필요 | 30%+ 이면 L1 교육 부족 |
| L2 → Engineering | 시니어 → 엔지니어링 | 버그, 인프라, 코드 변경 | 15%+ 이면 제품 안정성 문제 |
| L2 → Product | 시니어 → 프로덕트 | 기능 갭, 설계 결정 | 빈도 높으면 로드맵 괴리 |
| Any → Security | 모든 티어 → 보안 | 데이터 노출, 비인가 접근 | 발생 건수 자체가 위험 지표 |
| Any → Leadership | 모든 티어 → 경영진 | 고가치 고객 이탈, SLA 위반, 정책 예외 | 빈도 증가 시 구조적 문제 |

## Escalation Criteria

### 즉시 에스컬레이션
- 보안: 계정 탈취, 데이터 유출
- 법률/컴플라이언스
- 시스템 전체 장애
- 경영진 지정 요청

### 1차 시도 후 에스컬레이션
- 기술적: 트러블슈팅 후 미해결, 코드 수정 필요
- 복잡도: 지원팀 범위 초과, 커스텀 구현
- 영향도: 다수 고객, 프로덕션 다운
- 비즈니스: SLA 위반, 기준 초과 환불/크레딧
- 패턴: 동일 이슈 3명+ 보고, 재발

### 불필요 에스컬레이션 (지원팀 처리 가능)
- 문서화된 해결책 존재
- 설정/셋업으로 직접 해결 가능
- 안내/교육이 필요한 경우
- 이전 동일 티켓이 L1에서 해결된 이력

## Common Ticket Categories

| category | 특성 | 분석 시 주의점 |
|----------|------|---------------|
| 계정/인증 | FCR 높음, 해결시간 짧음, 자동화 적합 | FCR 낮으면 셀프서비스 UX 문제 |
| 청구/결제 | 민감도 높음, 재접수율 높을 수 있음 | 재접수 사유가 '상태 확인'이면 셀프서비스 부재 |
| 기술 문제/버그 | 에스컬레이션율 높음, 해결시간 길음 | L2→Eng 비율로 제품 안정성 판단 |
| 사용 방법 | 교육·가이드로 해결 가능 | 비율 높으면 온보딩·문서화 부족 |
| 기능 요청 | Product 에스컬레이션 대상 | 반복 요청은 로드맵 갭 지표 |
| 일반 문의 | FCR 높음, CSAT 높음 | 비중 높으면 FAQ·헬프센터 부족 |

## Operational Patterns

| 패턴 | 설명 |
|------|------|
| 카테고리 양극화 | 계정/인증·일반 문의는 양호, 기능 오류·결제/환불은 심각 — 10배 격차 |
| 해결시간 = 대기 + 처리 | 느린 카테고리 분해 시 배정 대기가 절반 이상 |
| 에스컬레이션 ≈ 만족도 하락 | 에스컬레이션 건의 CSAT가 1.0~1.5점 낮음 |
| 셀프서비스 부재 → 반복 접수 | 결제/환불 재접수의 40%+가 '상태 확인' |
| 상담원 편차 = 교육 기회 | 5%p+ 편차면 해결 패턴 표준화로 개선 가능 |

## Common Decisions

| decision | required_info |
|----------|--------------|
| 셀프서비스 확대 범위 | 카테고리별 FCR, 반복 문의 사유 Top 10, FAQ 커버리지 |

| 심각도 분류 재설계 | 분류 정확도, 심각도별 해결시간 분포, 오분류율 |
