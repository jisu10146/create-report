### 채널별 응답시간 SLA 벤치마크

채널별 데이터가 있을 때, 응답시간·해결시간의 적정 수준을 판단하는 기준.

| channel | 첫 응답 목표 | 해결 목표 | 분석 시 활용 |
|---------|------------|----------|-------------|
| 채팅 (Live Chat) | 2분 | 15분 | 채팅 첫 응답 2분 초과 시 인력 부족 또는 라우팅 지연 의심 |
| 이메일 | 4시간 | 24시간 | 이메일 해결 24h 초과 비율이 채널 SLA 위반의 주 원인인지 확인 |
| 전화 | 즉시 (대기 없음) | 10분 | 통화 중 해결 못 하고 후속 티켓 전환 비율이 높으면 에이전트 역량 문제 |
| 소셜 미디어 | 1시간 | 4시간 | 소셜 응답 지연은 브랜드 리스크로 직결 — Reputation 영향도와 함께 평가 |

## Terminology

| term | definition |
|------|-----------|
| SLA (Service Level Agreement) | 서비스 수준 계약. 티켓 유형/우선순위별 첫 응답·해결 목표 시간을 정의 |
| CSAT (Customer Satisfaction Score) | 고객 만족도. 보통 1-5점 또는 1-10점 척도. 해결 직후 설문으로 수집 |
| FCR (First Contact Resolution) | 1차 해결율. 에스컬레이션 없이 첫 번째 상담에서 해결된 비율. 높을수록 효율적 |
| MTTR (Mean Time to Resolution) | 평균 해결시간. 티켓 생성~종료까지 걸린 시간. 업무시간 기준과 캘린더 기준 구분 필요 |
| FRT (First Response Time) | 첫 응답시간. 티켓 생성~첫 에이전트 응답까지 걸린 시간 |
| Escalation | 에스컬레이션. 1차 상담원이 해결 불가 시 상위 담당자/팀으로 이관하는 프로세스 |
| Backlog | 미해결 잔여 티켓. 일정 시점의 미처리 티켓 수. 증가 추세면 인력 부족 신호 |
| Ticket Deflection | 셀프서비스(FAQ, 봇)로 티켓 접수 없이 해결된 비율 |
| Reopen Rate | 재오픈율. 해결 처리 후 고객이 다시 문의를 재개한 비율. 높으면 해결 품질 문제 |
| NPS (Net Promoter Score) | 순추천지수. 추천 의향 0-10점 중 추천자(9-10) 비율 - 비추천자(0-6) 비율 |
| CES (Customer Effort Score) | 고객 노력 점수. 문제 해결까지 고객이 들인 노력. 1(매우 쉬움)-7(매우 어려움) |
| Ticket Volume Trend | 티켓 유입량 추이. 주별/월별 패턴으로 인력 배치와 시즌 대비에 활용 |
| HEAR Method | 고객 응대 프레임워크. Hear(경청)→Empathize(공감)→Assess(평가)→Resolve(해결). 응대 품질 평가 기준으로 활용 |
| Channel Mix | 채널 구성 비율. 채팅/이메일/전화/소셜 등 채널별 티켓 유입 비중. 채널별 CSAT·해결시간 차이 분석에 활용 |

## Escalation Tiers

에스컬레이션 데이터를 분석할 때 각 티어의 적정 기준과 의미를 판단하는 데 활용.

| tier | from → to | when | 분석 시 체크포인트 |
|------|-----------|------|--------------------|
| L1 → L2 | 프론트라인 → 시니어/기술 지원 | 심화 조사, 전문 제품 지식, 고급 트러블슈팅 필요 | L1→L2 비율이 30%+ 이면 L1 교육 부족 또는 분류 기준 문제 |
| L2 → Engineering | 시니어 지원 → 엔지니어링 | 버그 확인, 인프라 이슈, 코드 변경 필요, 시스템 수준 조사 | L2→Eng 비율이 15%+ 이면 제품 안정성 문제 의심 |
| L2 → Product | 시니어 지원 → 프로덕트 | 기능 갭, 설계 결정 필요, 워크플로 불일치, 우선순위 결정 | L2→PM 빈도 높으면 고객 기대와 제품 로드맵 괴리 |
| Any → Security | 모든 티어 → 보안팀 | 데이터 노출, 비인가 접근, 취약점 보고, 컴플라이언스 | 정상 티어 무시하고 즉시 에스컬레이션 — 발생 건수 자체가 위험 지표 |
| Any → Leadership | 모든 티어 → 경영진 | 고가치 고객 이탈 위험, SLA 위반, 교차 기능 의사결정, 정책 예외, PR/법적 리스크 | 빈도 증가 시 구조적 프로세스 문제 신호 |

## Escalation Criteria

에스컬레이션율 분석 시, 적정 에스컬레이션과 불필요 에스컬레이션을 구분하는 기준.

### 즉시 에스컬레이션 (1차 시도 없이)
- 보안 문제: 계정 탈취, 데이터 유출, 비인가 접근
- 법률/컴플라이언스 이슈
- 위협 또는 괴롭힘 상황
- 경영진 지정 요청
- 시스템 전체 장애

### 1차 시도 후 에스컬레이션
- **기술적**: 일반 트러블슈팅 후에도 미해결, 버그 확인 후 코드 수정 필요, 인프라 조사 필요, 데이터 손상/유실
- **복잡도**: 지원팀 진단 범위 초과, 지원팀에 없는 접근 권한 필요, 커스텀 구현 관련
- **영향도**: 다수 고객 영향, 프로덕션 다운, 데이터 무결성 위험
- **비즈니스**: 고가치 고객 위험, SLA 위반 임박/발생, 고객이 경영진 개입 요청, 기준 금액 초과 환불/크레딧 요청, 복잡한 청구 분쟁
- **시간**: SLA 초과 상태, 고객 대기 시간 과도, 일반 채널로 진전 없음
- **패턴**: 동일 이슈 3명+ 고객 보고, 수정 완료로 알려진 이슈 재발, 시간에 따라 심각도 증가

### 지원팀에서 처리 가능한 경우 (불필요 에스컬레이션)
- 문서화된 해결책 또는 알려진 워크어라운드 존재
- 설정/셋업 이슈로 지원팀이 직접 해결 가능
- 고객에게 안내/교육이 필요한 경우 (수정이 아님)
- 알려진 제한 사항으로 대안이 문서화됨
- 이전 유사 티켓이 지원 레벨에서 해결된 이력

## Severity Classification

티켓 우선순위/심각도 데이터를 분석할 때 분류 정확도를 평가하는 기준.

| severity | definition | 적정 응답 기준 | 팔로업 주기 |
|----------|-----------|---------------|------------|
| Critical | 프로덕션 다운, 데이터 위험, 보안 침해, 다수 고가치 고객 영향 | 즉시 대응 | 내부 2시간 / 고객 2-4시간 |
| High | 주요 기능 장애, 핵심 고객 차단, SLA 위험 | 당일 대응 | 내부 4시간 / 고객 4-8시간 |
| Medium | 워크어라운드가 있는 중요 이슈, 긴급하지 않은 비즈니스 영향 | 금주 내 대응 | 내부 일 1회 / 고객 1-2영업일 |

## Business Impact Dimensions

에스컬레이션·이탈·SLA 위반 데이터를 해석할 때, 영향도를 다차원으로 평가하는 프레임워크.

| dimension | 분석 시 질문 |
|-----------|-------------|
| Breadth | 몇 명의 고객/유저가 영향받는가? 증가 추세인가? |
| Depth | 완전 차단 vs 불편 수준? 핵심 워크플로 영향 여부? |
| Duration | 얼마나 지속되었는가? 임계점까지 얼마나 남았는가? |
| Revenue | 위험에 처한 ARR 규모? 진행 중인 딜에 영향? |
| Reputation | 외부 공개 가능성? 레퍼런스 고객인가? |
| Contractual | SLA 위반 여부? 계약상 의무 사항 존재? |

## Common Ticket Categories

티켓 카테고리 분포를 분석할 때, 일반적인 CS 티켓 유형과 각 유형의 특성.

| category | 특성 | 분석 시 주의점 |
|----------|------|---------------|
| 계정/인증 (비밀번호 재설정 등) | FCR 높음, 해결시간 짧음, 자동화 적합 | FCR 낮으면 셀프서비스 UX 문제 의심 |
| 청구/결제 | 민감도 높음, 재접수율 높을 수 있음, 정확한 금액·일자 정보 필요 | 재접수 사유가 '상태 확인'이면 셀프서비스 부재 신호 |
| 기술 문제/버그 | 에스컬레이션율 높음, 해결시간 길음, 재현 단계 필요 | 에스컬레이션→엔지니어링 비율로 제품 안정성 판단 |
| 사용 방법/How-to | 교육·가이드로 해결 가능, FCR 중간 | 비율 높으면 온보딩·문서화 부족 신호 |
| 기능 요청 | 직접 해결 불가, Product 에스컬레이션 대상 | 반복 요청은 제품 로드맵 갭 지표 |
| 일반 문의 | FCR 높음, CSAT 높음 | 비중 높으면 정보 접근성 문제 (FAQ·헬프센터 부족) |

## Report Reader Priorities

CS 리포트를 보는 관리자가 실제로 확인하는 우선순위. 리포트 섹션 순서 설계 시 이 순서를 따름.
(출처: Zendesk/Freshdesk/Sprinklr/SuperOffice 2025-2026 CS Report Best Practices 리서치)

| 순위 | 관리자가 보는 것 | 핵심 질문 | 적합 컴포넌트 |
|------|-----------------|----------|-------------|
| 1 | 전체 건강 상태 | "지금 괜찮은가?" | ExecutiveSummary |
| 2 | KPI 벤치마크 대비 | "정상 범위인가?" | MetricHighlight |
| 3 | SLA 준수 현황 | "약속을 지키고 있는가?" | DataTable (카테고리×SLA) |
| 4 | 처리 속도 병목 | "어디서 느린가?" | StackedBarChart (대기 vs 처리) |
| 5 | 에이전트 성과 | "누가 도움이 필요한가?" | DataTable (상위 vs 하위) |
| 6 | 고객 만족도/NPS | "고객이 어떻게 느끼는가?" | StackedBarChart (NPS 3구간) |
| 7 | 채널별 성과 | "어느 채널이 나쁜가?" | DataTable (채널×지표) |
| 8 | 근본 원인 | "왜 이런 상황인가?" | InsightCard |
| 9 | 개선 로드맵 | "어떻게 해결하는가?" | StrategyTable |

주의:
- 채널 간 편차가 작으면(NPS 편차 5p 이내) 채널 섹션은 Persona Critic에서 제거될 수 있음
- 퍼널은 해결율 90%+ 이면 drop-off가 안 보여 의미 없음 — 병목이 "건수 감소"가 아닌 "시간 지연"이면 StackedBarChart가 적합
- 에이전트 편차가 5%p 이내면 "교육 여지 존재" 수준, 10%p+ 이면 "구조적 교육 문제"

## NPS Analysis Guide

NPS 데이터가 있을 때의 분석 가이드.

| 항목 | 기준 | 해석 |
|------|------|------|
| NPS +30~50 | 양호 (SaaS 기준) | 현재 상태 유지, 추천자 확대 전략 |
| NPS 0~+30 | 개선 필요 | 중립자→추천자 전환에 집중 |
| NPS -50~0 | 심각 | 비추천자 원인 분석 우선, 카테고리별 분해 필수 |
| NPS -50 이하 | 위기 | 해당 카테고리/채널 즉시 개선 필요 |

분석 방법:
- 전체 NPS만으로는 부족 — 반드시 카테고리별·채널별로 분해하여 어디서 비추천자가 발생하는지 특정
- NPS 점수와 CSAT를 교차 비교 — NPS는 추천 의향(미래 행동), CSAT는 직전 경험 만족도(과거 행동)
- 비추천자 비율 72%+ 인 카테고리가 있으면, 해당 카테고리가 전체 NPS를 끌어내리는 단일 최대 요인일 가능성 높음
- 에스컬레이션 발생 건의 NPS는 미발생 건보다 통상 20~40점 낮음 — 에스컬레이션 감소가 NPS 개선의 가장 빠른 경로

## Operational Patterns Learned

실제 CS 데이터 분석에서 반복적으로 발견되는 패턴. 리포트 설계 시 참고.

| 패턴 | 설명 | 리포트에서 다루는 방법 |
|------|------|---------------------|
| 카테고리 양극화 | 계정/인증·일반 문의는 양호(CSAT 4.2+), 기능 오류·결제/환불은 심각(CSAT 3.4 이하) — 같은 CS팀 내 10배 격차 | SLA DataTable에서 카테고리 교차 비교로 격차 시각화, InsightCard에서 원인별(프로세스/시스템/역량) 분리 |
| 해결시간 = 대기 + 처리 | 느린 카테고리의 해결시간 내부를 분해하면 "배정 대기"가 절반 이상인 경우가 많음 | StackedBarChart로 카테고리별 대기 vs 처리 분해 |
| 에스컬레이션 ≈ 만족도 하락 | 에스컬레이션된 건의 CSAT는 미발생 건보다 1.0~1.5점 낮음 | MetricHighlight에서 에스컬레이션 건/미발생 건 CSAT 비교 |
| 셀프서비스 부재 → 반복 접수 | 결제/환불 카테고리에서 "상태 확인" 재접수가 재접수의 40%+ 차지 | InsightCard에서 "시스템" 차원 원인으로 진단 |
